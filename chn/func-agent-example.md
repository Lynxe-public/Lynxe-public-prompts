# 函数化Agent示例

## 功能说明

**Func-Agent 的核心就是用函数形式来定义 Agent。** 一个 Func-Agent 可抽象为四部分：

- **函数名**：唯一标识，便于调用与路由。
- **结构化参数表**：入参以 JSON 描述（如参数1、参数2、参数3），而非纯自然语言。
- **函数体**：用文字描述要完成的任务，由 Agent 执行；函数体内可调用其他 Func-Agent。
- **结构化返回值**：结果以 JSON 等形式返回，便于下游系统或其它 Agent 使用。

例如本示例中的「技术专家处理问题」：函数名为该名称；入参可为 `{"question": "如何优化这条 SQL"}`；函数体描述为「针对用户的技术类问题给出专业解答」并由 Agent 执行；返回结构化的解答内容，供路由函数或工单系统使用。

可以理解为：给 Agent 贴了一个「函数形式的标签」，使其能被非自然语言、结构化地调用，从而在系统集成和 Agent 间协作时提高调用的准确性。

**这样做的两个直接好处：** 一是既有系统的请求都可以转换为 JSON，接入非常简单；二是输入输出都是结构化的，Agent 之间协作时传递的调用信息更准确，不会因为自然语言歧义导致调用错误。

本示例演示如何用函数组合和路由机制实现智能问题分类与专家系统。你可以学到如何创建多个专业化 Agent 函数、如何做函数间组合调用，以及如何通过路由 Agent 把问题分配给合适的专家。本例中的技术专家、数据库专家和业务处理专家对应三个子 Agent，他们拥有完全独立的记忆资源并可并行使用，是当前我们实践中推荐的 sub-agent 使用方式。

## 使用流程

**版本要求**: 4.8.6（含）以上

**第1步** [打开Lynxe系统](https://github.com/spring-ai-alibaba/Lynxe)

**第2步** 导入数据
- 导入 [prompt-script/func-agent-example.json](../prompt-script/func-agent-example.json) 文件中的计划模板
- 系统会自动识别并加载所有函数Agent

**第3步** 理解函数结构
- **技术专家处理问题**: 专门处理技术类问题的Agent函数
- **数据库专家处理问题**: 专门处理数据库类问题的Agent函数
- **业务专家处理问题**: 专门处理业务类问题的Agent函数
- **主入口-路由方法**: 根据问题类型自动路由到相应专家的主入口函数 这是最核心的启动函数，你也需要确保他的正确的引用了其他三个func—agent，这应该是自动的，但有可能因为版本不同而存在被错误忽略的问题

**第4步** 执行路由函数
- 运行"主入口-路由方法"函数
- 输入用户需求或问题
- 系统会自动判断问题类型并路由到相应的专家函数
- 专家函数处理后会返回相应的答案


## 预期结果

- 成功导入4个函数Agent：路由函数和3个专家函数
- 路由函数能够正确识别问题类型（技术/数据库/业务）
- 问题被自动路由到对应的专家函数进行处理
- 各专家函数返回符合其专业领域的答案
- 支持函数组合和复用，体现"函数是第一公民"的设计理念

## 技术要点

- Function-Agent能力：智能函数组合和自动任务分解，这也是整个系统最关键的特征之一
- 函数组合和复用：支持将复杂任务分解为可复用的工具函数
- 路由机制：根据问题类型自动选择最合适的处理函数
- 函数发布：所有函数都可以作为独立的工具被调用
- 参数传递：支持函数间的参数传递和结果返回
